# 辩证协议（内核，领域无关）

本协议是 Crucible 引擎内核的完整规范。内核只拥有四样东西：角色协议、循环控制流、收敛判断、认知增益的记录结构。任何领域知识（前提类型、完整性约束、方法市场）都来自已加载的实例文件——本协议不含任何领域特定内容。

## 一、三个角色

| 角色 | 职责 | 提问方式 | 产出 |
|---|---|---|---|
| **倡议者**（Proposer） | 构建论证、提出主张、建立解释框架 | "最好的解释是什么？" | 命题、论证链、支持性前提 |
| **挑战者**（Challenger） | 攻击论证薄弱环节、暴露隐藏假设、寻找反例 | "这个论证在什么地方最脆弱？" | 驳斥记录、被暴露的隐藏前提、反例 |
| **反思者**（Reflector） | 元层面审视论证框架本身 | "我们是不是在问错误的问题？" | 框架转换建议、认知增益、未解决张力 |

### 倡议者规范

- 每个命题是一句明确、可被攻击、可证伪的主张，不含糊其辞
- 论证链每一步显式可检验，不允许逻辑跳跃
- 声明依赖的**全部**前提，包括原本隐含的假设
- 从实例的方法市场中选择合适方法并注明 id
- 遵守实例的全部完整性约束

### 挑战者规范

攻击手段：反例、隐藏前提、逻辑跳跃、概念偷换、替代解释、边界条件、完整性约束违规。

判定纪律（协议的心脏，逐条执行）：

1. 对每个存活命题分别评估，给出攻击列表（每条标注攻击类型与严重度 fatal/serious/minor）与判定
2. 只有攻击确实动摇**论证核心**（而非措辞或次要细节）时，才判定"实质性削弱"
3. 任何违反实例中 severity=blocker 完整性约束的行为，直接构成实质性削弱
4. **不要为攻击而攻击**：找不到新的有效攻击角度时，如实判定"未被削弱"并说明理由——这正是收敛信号；重复上一轮已被消化的攻击不算新角度
5. 判定必须附理由（verdict_reason）

### 挑战者的独立上下文（强烈建议）

共享上下文的挑战者容易被倡议者的框架同化。若环境支持派发子代理，**必须**用 `crucible-challenger` 子代理执行攻击，输入只包含：

- 待辩证的问题
- 各命题的编号、主张、论证链、声明的前提
- 实例的完整性约束与前提类型（可直接粘贴实例文件相关段落）
- 用户提供的前提/数据

**不传**倡议者的探索过程、犹豫、或"我觉得这个论证很强"之类的评价。子代理返回逐命题的评估。

无子代理能力时在同一上下文换帽子，此时额外自检：写攻击前先问自己"如果这是别人交给我的命题，我会从哪里下手？"；禁止在攻击文本中为命题辩护。

### 修正阶段规范（倡议者，被削弱后）

诚实面对攻击，二选一：

- **revise**：命题可在吸收攻击后修正（收窄范围、补上前提、修补论证链）→ 给出修正后的命题与论证链。禁止措辞游戏假装修正。
- **abandon**：攻击致命、命题核心无法挽救 → 果断放弃，记入被驳斥名单。

无论哪种，都必须写下 **lesson**（这次驳斥教会我们什么），教训传给后续所有倡议者行动。

### 反思者规范

不参与具体攻防，只在元层面审视：

- 框架评价：问题问对了吗？前提类型合适吗？方法用对了吗？
- 认知增益：本轮之前不知道的 X、被推翻的假设 Y、新暴露的问题 Z
- 未解决张力：当前框架无法解决的矛盾或悖论
- 框架转换：只有框架确实需要转换时才给出重构后的问题
- 判定纪律：只有确实发现新的意外或框架转换机会时才申报"有新发现"；连续没有新发现是收敛信号，如实报告

## 二、循环流程

每轮依次执行：

1. **倡议者提出**（仅首轮、全部命题被驳斥后、或框架转换后）：针对当前问题提出 1-3 个命题
2. **挑战者攻击**：攻击全部存活命题，逐个判定
3. **修正阶段**：每个被实质性削弱的命题 → revise 或 abandon
4. **反思者审视**：框架评价、认知增益、张力、可能的框架转换
5. **状态行 + 收敛判断**（见下）

命题强度阶梯（随对抗自动演化）：

- 连续 0 轮未被削弱 = `speculative`（新提出或刚修正）
- 连续 1 轮 = `tentative`
- 连续 ≥2 轮 = `robust`
- 修正后归零重计——修正后的命题需要重新经受攻击

框架转换时：当前问题替换为重构后的问题，下一轮倡议者针对新问题重新提出命题；已幸存的命题保留（它们的幸存记录仍然有效），但最终置信度降为 conditional。

## 三、收敛纪律

每轮结束**必须**输出状态行（让循环有账可查，不可省略）：

```
—— 轮次 k ｜ 实质性削弱：是/否 ｜ 反思者新发现：是/否 ｜ 无削弱连击 a/N ｜ 无新发现连击 b/M ——
```

- 本轮有任何命题被实质性削弱 → a 归零，否则 a+1
- 反思者申报新发现或发生框架转换 → b 归零，否则 b+1
- **收敛**：a ≥ N 且 b ≥ M → 终止，termination=converged
- **硬上限**：达到等级的轮数上限仍未收敛 → 终止，termination=max_rounds（防止无限循环）
- N/M 与硬上限由深度等级决定（见 SKILL.md 的 L1-L4 表）

## 四、统一输出结构

无论哪个领域实例，最终交付都是这个形状（认知审计追踪：后来者要能追溯"结论怎么来的、经历过什么攻击、什么被排除过"）：

```markdown
## 幸存的命题
- [强度 robust/tentative/speculative] 命题内容
  - 论证链：…
  - 依赖前提：…
  - 幸存于哪些攻击：…（逐条）
  - 修正次数：…

## 被驳斥的命题
- 命题内容
  - 驳斥：…
  - 教训：这次驳斥教会我们什么

## 认知增益
- 之前不知道的 X / 被推翻的假设 Y / 新暴露的问题 Z

## 未解决的张力
- 当前框架无法解决的矛盾或悖论

## 框架评价
反思者对分析框架本身的最终评价

## 置信度
overall | conditional | low —— 及理由
```

置信度判定：

- `overall`：自然收敛 + 至少一个 robust 幸存命题 + 未发生框架转换
- `conditional`：自然收敛但无 robust 命题，或运行中发生过框架转换（结论依赖新框架成立）
- `low`：撞硬上限仍未收敛（对抗未穷尽），或没有命题幸存

用户要求机器可读输出时，同样内容以 JSON 给出，字段名：`surviving_propositions`（内含 `proposition/argument_chain/premises_used/methods_used/challenges_survived/revisions/strength`）、`refuted_propositions`（`proposition/refutation/lesson`）、`epistemic_gains`、`unresolved_tensions`、`framework_critique`、`confidence`、`termination`、`rounds`。此结构与本仓库 Python 引擎（`crucible run --output`）的输出一致。
