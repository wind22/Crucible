# 辩证引擎：通用推理架构

本文档描述一个**通用辩证推理引擎**（Dialectical Engine），它是 `data-analysis-skills` 认知层架构的抽象和推广。核心思想是：同一个辩证内核，加载不同的前提类型和完整性约束，可以完成从数据分析到纯粹思辨的多种论证任务。

## 一、核心命题

> **一个辩证引擎，多种领域实例。区别仅在于"前提是什么"和"什么算错误"。**

```
                    ┌─────────────────────────┐
                    │     辩证引擎内核          │
                    │  ┌───────────────────┐  │
                    │  │ Proposer          │  │
                    │  │ Challenger        │  │
                    │  │ Reflector         │  │
                    │  │ Convergence Gate  │  │
                    │  └───────────────────┘  │
                    └──────────┬──────────────┘
                               │
            ┌──────────────────┼──────────────────┐
            │                  │                  │
            ▼                  ▼                  ▼
    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
    │ 数据分析实例  │  │ 纯思辨实例    │  │ 战略推演实例  │
    │              │  │              │  │              │
    │ 前提：数据    │  │ 前提：概念    │  │ 前提：混合    │
    │ 约束：数据    │  │ 约束：逻辑    │  │ 约束：混合    │

    └──────────────┘  └──────────────┘  └──────────────┘
```

## 二、通用内核

辩证引擎的内核是一组**领域无关**的认知协议。它不关心你在论证什么，只关心论证的质量。

### 2.1 三个角色

| 角色                           | 职责                                       | 提问方式                     | 产出                                       |
| ------------------------------ | ------------------------------------------ | ---------------------------- | ------------------------------------------ |
| **倡议者**（Proposer）   | 构建论证、提出主张、建立解释框架           | "最好的解释是什么？"         | 命题、论证链、支持性前提                   |
| **挑战者**（Challenger） | 攻击论证的薄弱环节、暴露隐藏假设、寻找反例 | "这个论证在什么地方最脆弱？" | 驳斥记录、被暴露的隐藏前提、反例           |
| **反思者**（Reflector）  | 元层面审视论证框架本身                     | "我们是不是在问错误的问题？" | 框架转换建议、认知增益记录、未解决张力清单 |

三个角色可以是同一模型的不同 prompt，也可以是不同模型。能力强时用同一模型切换角色即可，能力极强时甚至可以用更弱的模型做挑战者（不对称对抗往往更有价值——用简单问题攻击复杂论证）。

### 2.2 辩证循环

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  倡议者提出命题 + 论证链                           │
│          │                                       │
│          ▼                                       │
│  挑战者攻击：反例？隐藏前提？逻辑跳跃？替代解释？     │
│          │                                       │
│          ├── 被实质性削弱 ──→ 倡议者修正或放弃       │
│          │                                       │
│          ├── 未被削弱 ──→ 命题强度提升              │
│          │                                       │
│          ▼                                       │
│  反思者审视：框架对吗？有什么意外发现？               │
│          │                                       │
│          ├── 新问题或框架转换 ──→ 重新开始           │
│          │                                       │
│          └── 无新发现 ──→ 进入收敛判断               │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 2.3 收敛条件

循环不因"流程走完"而终止，终止条件来自对抗的质量：

- **主要条件**：连续 N 轮挑战未产生实质性削弱（N 随能力等级调整，默认 2）
- **辅助条件**：反思者连续 M 轮未发现新的意外或框架转换机会
- **硬上限**：达到预设的深度上限（防止无限循环，尤其是弱模型场景）
- **时间上限**：认知增益递减时自动终止（"再想下去边际收益为负"）

### 2.4 输出

无论哪个领域实例，辩证引擎的输出结构是统一的：

```
{
  "surviving_propositions": [
    {
      "proposition": "...",
      "argument_chain": [...],
      "challenges_survived": [...],
      "strength": "robust | tentative | speculative"
    }
  ],
  "refuted_propositions": [
    {
      "proposition": "...",
      "refutation": "...",
      "lesson": "这次驳斥教会我们什么"
    }
  ],
  "epistemic_gains": [
    "之前不知道的 X",
    "被推翻的假设 Y",
    "新暴露的问题 Z"
  ],
  "unresolved_tensions": [
    "矛盾或悖论，当前框架无法解决"
  ],
  "framework_critique": "反思者对分析框架本身的评价",
  "confidence": "overall | conditional | low — 及理由"
}
```

这个输出结构本身就是一种**认知审计追踪**——后来者可以追溯"这个结论是怎么来的，它经历过什么攻击，有什么东西被排除过"。

## 三、领域实例：加载不同的前提和约束

辩证引擎的内核是通用的。要将其实例化到特定领域，需要加载三样东西：

1. **前提类型声明**：什么算有效的输入
2. **完整性约束**：什么算不可接受的错误
3. **方法市场**：该领域已知的论证或分析工具

### 3.1 数据分析实例

```
前提类型：经验数据（来自外部世界的、可重复验证的观察记录）
完整性约束：
  - 数字来自可追溯的源数据
  - 逻辑闭合（分子分母可加总、双算检测）
  - 呈现不误导（格式合规、限制声明）
方法市场：
  - 贡献度分解（additive/LMDI/ratio/Shapley）
  - 同期群分析（经典/区间/滚动留存、转化、LTV）
  - 统计检验（显著性、置信区间、功效分析）
  - 因果推断（DID、RDD、IV、匹配方法）
  - 时间序列分解（趋势、季节、残差）
  - 异常检测（统计过程控制、隔离森林）
```

### 3.2 纯思辨实例

```
前提类型：概念定义、公理、思想实验、先验命题
完整性约束：
  - 前提被显式声明（不隐藏假设）
  - 逻辑不自我矛盾
  - 概念使用一致（无定义漂移）
  - 推理步骤显式化
  - 引用链可追溯
方法市场：
  - 归谬法（假设命题为真 → 推出矛盾 → 命题为假）
  - 思想实验（构造情景，检验直觉或原则）
  - 概念谱系追溯（概念的起源与演化，识别语义漂移）
  - 苏格拉底式诘问（持续追问"你这么说到底是什么意思"）
  - 类比推理验证（A:B::C:D，检验结构同构性）
  - 边界条件探索（命题在极端情况下还成立吗）
  - 反事实推演（如果 X 不成立，会怎样）
  - 框架对比（同一问题用不同理论框架分析，差异在哪）
  - 前提敏感性测试（改变某个前提 → 结论如何变化）
```

### 3.3 战略推演实例

```
前提类型：数据 + 假设 + 竞争性预判 + 约束条件
完整性约束：
  - 数据部分：同数据分析实例
  - 逻辑部分：同纯思辨实例
  - 额外：假设被显式标注为假设（不与数据混淆）
  - 额外：竞争性预判被显式标注（"对手可能做 X" vs "我们知道 X"）
方法市场：
  - 情景规划（best/base/worst case，带触发条件）
  - 博弈矩阵（参与者、策略、收益、均衡）
  - 预判 vs 反预判（"对手预判了我们的预判"的递归）
  - 稳健性分析（计划在哪些假设变化下仍成立）
  - 期权思维（延迟决策的价值、不可逆决策的成本）
  - 竞争态势推演（波特五力、价值网络）
  - 系统动力学（反馈回路、延迟效应、非线性）
```

### 3.4 其他可能的实例

| 领域               | 前提类型                       | 核心约束                         | 典型方法                                           |
| ------------------ | ------------------------------ | -------------------------------- | -------------------------------------------------- |
| **法律论证** | 法条、判例、事实认定、法理     | 举证责任、证据可采性、程序正义   | 类比推理（遵循先例）、文义/体系/目的解释、比例原则 |
| **技术决策** | 需求、约束、技术事实、成本     | 可验证性、可复现性               | 权衡分析、ADR（架构决策记录）、失败模式分析        |
| **学术评审** | 论文、数据、方法论、文献       | 可复现性、同行验证               | 方法论审查、结论强度校准、贡献新颖性判断           |
| **历史解释** | 史料、档案、考古证据、口述     | 来源可靠性、时代语境、反事实谨慎 | 多重证据交叉验证、反事实约束、时代错置检测         |
| **伦理判断** | 价值原则、情境事实、利益相关方 | 普遍性检验、权利冲突显式化       | 原则权衡、滑坡论证检测、利益相关方视角切换         |

## 四、内核与实例之间的接口

要让内核真正可插拔，需要定义清晰的接口：

### 4.1 实例描述符

```json
{
  "domain": "pure-reasoning",
  "premise_types": [
    {
      "type": "conceptual_definition",
      "description": "概念定义，必须显式声明，不可在论证中偷换",
      "validation": "challenger 检查概念使用一致性"
    }
  ],
  "integrity_rules": [
    {
      "id": "no_hidden_premises",
      "description": "所有前提必须显式声明",
      "checker": "challenger",
      "severity": "blocker"
    },
    {
      "id": "no_self_contradiction",
      "description": "不能同时主张 A 和 非A",
      "checker": "challenger + formal_logic",
      "severity": "blocker"
    }
  ],
  "method_market": [
    {
      "id": "reductio_ad_absurdum",
      "description": "归谬法",
      "when_to_use": "需要检验命题逻辑必然性时",
      "input_schema": { "proposition": "string" },
      "output_schema": { "contradiction_found": "bool", "chain": "string[]" }
    }
  ]
}
```

### 4.2 内核不拥有的东西

内核**不应该**拥有：

- 任何领域特定的方法（方法是市场里的条目）
- 任何领域特定的约束（约束是实例描述符的一部分）
- 任何领域特定的输出格式（输出结构是通用的，渲染是下游的事）

内核**只**拥有：

- 辩证循环的控制流
- 角色协议（proposer/challenger/reflector 的交互规范）
- 收敛判断逻辑
- 认知增益的记录结构

## 五、能力自适应：同一个引擎，不同深度

内核的质量不随模型能力自动变好，但内核的**使用深度**随模型能力自动变深：

| 能力等级   | 辩证深度      | 角色分配                                                        | 收敛条件              | 典型场景           |
| ---------- | ------------- | --------------------------------------------------------------- | --------------------- | ------------------ |
| L1（弱）   | 1 轮攻击      | 同一模型切换角色                                                | 攻击完成即终止        | 快速 sanity check  |
| L2（中）   | 2-3 轮循环    | 同一模型切换角色                                                | 连续 2 轮无实质性削弱 | 日常分析           |
| L3（强）   | 收敛为止      | 可选不同模型                                                    | 收敛或硬上限          | 深度诊断、重要决策 |
| L4（极强） | 收敛 + 元对抗 | 建议不同模型（强模型做 proposer，弱模型做 challenger 也有价值） | 收敛 + 反思者满意     | 框架突破、范式反思 |

关键原则：**L1 的表现不应该比当前框架更差**（向后兼容），**L4 的潜力不应该被框架限制**（向前开放）。

## 六、从数据分析到辩证引擎的迁移路径

不需要重写。三步即可：

### Step 1：提取内核协议

把当前隐含在 `da-analysis-master` 中的认知流程（SCQ → MECE → 假设 → 合龙 → Guardrail）抽象为显式的辩证循环协议。新增一个 `references/dialectical-protocol.md`，定义角色、循环、收敛规则——这些都是领域无关的。

### Step 2：拆分地基与实例

把当前混在一起的"地基规则"和"分析方法"拆开：

- 地基规则（数字格式化、闭合验算、路径脱敏）→ 作为"数据分析实例"的 `integrity_rules` 加载
- 分析方法（贡献度、同期群）→ 作为"数据分析实例"的 `method_market` 加载
- 认知流程（辩证循环）→ 提升为通用内核，不依赖任何领域概念

### Step 3：注册第二个实例

以纯思辨为 second instance，验证内核的领域无关性。不需要完整实现——只需要用同一个内核跑一次纯思辨论证，证明内核可以不依赖数据而工作。

## 七、设计原则总结

1. **内核领域无关**：辩证循环不关心你在论证什么
2. **实例可插拔**：加载不同的前提类型、约束规则、方法市场即可切换领域
3. **地基永远在，但地基的形态随领域变化**：数据分析的地基是数据完整性，思辨的地基是逻辑完整性，战略推演的地基是两者的混合
4. **深度自适应**：同一个内核，模型越强，辩证越深，天花板越高
5. **认知审计追踪**：输出不只是结论，还包括"这个结论是怎么来的，经历过什么攻击"
6. **向后兼容**：当前的数据分析框架是辩证引擎在数据领域的第一个实例，不应被破坏
